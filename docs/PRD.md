# 학원 숙제 플랫폼 MVP PRD

## 1) 프로젝트 목표

### 문제 정의
- 오프라인 학원에서 숙제 전달, 제출 확인, 피드백 전달이 카카오톡/종이로 분산되어 관리 비용이 높다.
- 선생님은 "누가 제출했는지", "누가 미제출인지", "어떤 피드백을 줬는지"를 한 화면에서 보기 어렵다.
- 학생은 본인 숙제 목록과 피드백 이력을 체계적으로 확인하기 어렵다.

### 목표
- 선생님 중심의 숙제 등록/제출 관리/피드백 작성 프로세스를 웹에서 일원화한다.
- 학생이 숙제 열람 → 텍스트 답안 제출 → 피드백 확인까지 한 번에 처리할 수 있게 한다.
- MVP 단계에서 운영 가능한 최소 기능을 1차 릴리즈한다.

### 성공 지표 (MVP)
- 제출률: 등록된 숙제 대비 학생 제출 비율 주간 70% 이상
- 피드백 작성률: 제출 건 대비 피드백 작성 비율 80% 이상
- 핵심 플로우 완료율: 숙제 열람 → 제출 완료 전환율 85% 이상
- 운영 효율: 선생님이 제출 현황 확인에 쓰는 시간 주당 30% 단축(정성 지표)

## 2) 사용자 유형

### 선생님 (Teacher)
- 권한: 숙제 생성/수정/삭제, 과제 대상 학생 지정, 제출 현황 조회, 피드백 작성/수정
- 주요 니즈:
  - 빠른 숙제 등록
  - 미제출자 식별
  - 제출별 피드백 기록

### 학생 (Student)
- 권한: 본인에게 할당된 숙제 조회, 텍스트 답안 제출/수정(마감 전), 피드백 조회
- 주요 니즈:
  - 마감일 기준 숙제 확인
  - 간편 제출
  - 피드백 확인 및 학습 반영

### 관리자 (MVP에서는 내부 운영자 수준)
- MVP에서는 별도 UI 제공 없이 DB/콘솔 관리로 대체
- 역할: 계정 생성, 교사/학생 role 부여, 반(클래스) 매핑 초기 세팅

## 3) 핵심 기능

1. 인증/권한
- Supabase Auth 기반 로그인(이메일/비밀번호 또는 매직링크 중 하나 선택)
- role 기반 접근 제어(teacher, student)

2. 숙제 관리 (Teacher)
- 숙제 생성: 제목, 설명, 마감일, 대상 클래스/학생
- 숙제 목록/상세 조회
- 숙제 수정/마감 처리(필요 시)

3. 숙제 열람/제출 (Student)
- 본인에게 배정된 숙제 목록 조회
- 숙제 상세에서 텍스트 답안 제출
- 마감 전 답안 수정(최신본 1개 유지)

4. 제출 현황/피드백 (Teacher)
- 숙제별 제출 현황(제출/미제출, 제출시각)
- 학생 제출 상세 확인
- 텍스트 피드백 작성/수정

5. 피드백 확인 (Student)
- 본인 제출 건에 대한 피드백 목록/상세 확인

## 4) MVP 범위

### 포함
- 역할 분리된 로그인 및 기본 대시보드
- Teacher: 숙제 생성/조회, 제출 현황 조회, 피드백 작성
- Student: 숙제 조회, 텍스트 제출, 피드백 확인
- 반(Class) 단위 숙제 할당
- 제출 상태 계산(미제출/제출)

### 기술 범위
- 프론트: Next.js App Router + TypeScript + Tailwind + shadcn/ui
- 백엔드/BaaS: Supabase(Postgres + Auth + RLS)
- 배포: Vercel

## 5) 제외 기능 (MVP 이후)
- 파일 업로드(이미지/PDF/오디오) 제출
- 자동 채점, 루브릭, AI 피드백
- 푸시 알림/문자 알림
- 학부모 계정 및 학부모 뷰
- 다중 학원(멀티 테넌시)
- 출결/성적 통합 관리
- 실시간 채팅

## 6) 사용자 시나리오

### 시나리오 A: 선생님이 숙제를 등록한다
1) 선생님 로그인
2) `새 숙제 만들기` 클릭
3) 제목/설명/마감일/대상 클래스 선택 후 저장
4) 시스템이 대상 학생들에게 숙제를 할당 상태로 생성
5) 선생님은 숙제 상세에서 전체 대상/제출률 확인

### 시나리오 B: 학생이 숙제를 제출한다
1) 학생 로그인
2) `내 숙제` 페이지에서 마감 임박/미제출 숙제 확인
3) 숙제 상세 진입 후 답안 텍스트 작성
4) `제출` 클릭
5) 시스템이 제출 시각 저장, 상태를 제출로 갱신

### 시나리오 C: 선생님이 피드백을 작성한다
1) 선생님이 숙제 상세에서 제출자 목록 확인
2) 특정 학생 제출 건 클릭
3) 피드백 텍스트 입력 후 저장
4) 학생 화면에서 해당 제출 건의 피드백이 노출됨

### 시나리오 D: 학생이 피드백을 확인한다
1) 학생이 `피드백` 또는 숙제 상세 내 피드백 영역 진입
2) 선생님 피드백 텍스트 및 작성 시각 확인
3) 필요 시 숙제 재학습

## 7) 비기능 요구사항 (MVP 기준)
- 보안: 모든 데이터는 RLS 정책으로 role/소유권 기준 접근 제한
- 성능: 목록 페이지 첫 응답 2초 내(일반 네트워크 기준)
- 감사 추적: 생성/수정 시각(`created_at`, `updated_at`) 기록
- 사용성: 모바일 최소 대응(반응형), 데스크톱 우선
